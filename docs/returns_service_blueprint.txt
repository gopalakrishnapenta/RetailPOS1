==============================================
TECHNICAL BLUEPRINT: MULTI-SERVICE RETURN SYSTEM
==============================================

I. OVERVIEW
------------
The Return Service is built as a distributed transaction. It coordinates between Orders, Catalog, and Admin services to ensure data consistency.

II. KEY COMPONENTS
-------------------
1.  **RetailPOS.Contracts (Shared Library)**:
    - Defines `OrderReturnedEvent`.
    - Purpose: Standardizes the communication message for RabbitMQ.
  
2.  **OrdersService (Source Of Truth)**:
    - `ReturnService.cs`: Business logic for refund validation.
    - `OrderReturnedEvent`: Published to RabbitMQ when a return is "Approved".
  
3.  **CatalogService (Inventory Synchronizer)**:
    - `OrderReturnedConsumer.cs`: Listens for approved returns.
    - Action: `StockQuantity += ReturnedQuantity`.
  
4.  **AdminService (Financial Summary)**:
    - `DashboardService.cs`: Dynamically fetches approved returns via HTTP.
    - Logic: `Net Sales = Gross Sales - Total Refunds`.

III. DATA FLOW (STEP-BY-STEP)
------------------------------
[ Step 1: INITIATE ] 
Cashier calls `POST api/returns/initiate`.
Service: Records the return in the `Returns` table with Status="Initiated".

[ Step 2: APPROVE ]
Manager calls `POST api/returns/{id}/approve`.
Service: Validates original Bill, updates Status to "Approved", and publishes Event.

[ Step 3: ASYNC UPDATE ]
CatalogService consumes Event $\rightarrow$ Updates Product Stock.

[ Step 4: DASHBOARD ]
Manager views Dashboard $\rightarrow$ DashboardService fetches returns and subtracts them from Total Sales.

==============================================
Architecture Pattern: Event-Driven SEDA (Staged Event-Driven Architecture)
